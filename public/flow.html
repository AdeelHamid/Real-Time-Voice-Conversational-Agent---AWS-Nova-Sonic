How The Voice Flows Work

This page explains what happens in our modular approach and what happens in Nova Sonic realtime mode.

Modular Approach Flow (LLM + Polly + Services)

  1. User speaks in browser Browser captures microphone audio chunks and streams to backend.
  2. Speech to text stage Audio is transcribed into text for reasoning.
  3. LLM reasoning stage LLM generates response text based on prompt + context.
  4. Data and memory stage Context/cache can be read from DynamoDB, S3, vector retrieval, and cache layers.
  5. TTS stage (Polly) Response text is converted into audio.
  6. Audio playback to user Generated speech audio streams back to browser speaker.

Interrupt behavior: backend should stop current TTS stream and start new reasoning cycle on barge-in.

Nova Sonic Realtime Flow

  1. Session starts Browser opens WebSocket to Node backend and backend opens Bedrock bidirectional stream.
  2. User audio streaming Browser sends PCM audio chunks continuously.
  3. Realtime model processing Nova Sonic handles speech understanding + reasoning + speech generation in one stream.
  4. Assistant output streaming Text and audio output chunks are returned in realtime and played immediately.
  5. Barge-in handling If user starts speaking while assistant is talking, client sends interrupt and server starts a new stream epoch.
  6. Stale output protection Client ignores old stream chunks using stream epoch IDs, improving interrupt accuracy.

When You Speak vs When Nova Speaks

When You Speak

  • Mic audio is chunked and sent to backend.
  • Avatar switches to Listening mode.
  • If assistant is speaking, interrupt is triggered.
  • New stream session starts for your latest utterance.

When Nova Sonic Speaks

  • Backend receives realtime audio output chunks.
  • Browser queues audio buffers for smooth playback.
  • Avatar switches to Speaking mode.
  • Performance metrics capture first-response latency.
